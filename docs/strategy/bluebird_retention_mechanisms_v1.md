# Project BlueBird — 리텐션 메커니즘 제안

**문서 버전**: v1 (2026-04-30)
**대상 독자**: Claude Code, 파운더
**관련 문서**: `positioning-and-vision-v1.md`, `bluebird_competitive_strategy_v1.md`, `bluebird_stress_integration_v1.md`
**목적**: BlueBird의 리텐션 강화를 위한 두 가지 메커니즘(내 사용설명서, 익명 통계 맥락화) 제안. 본 문서는 streak·유저 상호작용·게이미피케이션 같은 "표류 경로" 대신, 분석가 톤·자율성 미션·차별화 축과 정합하는 리텐션 동력을 정의한다. Claude Code는 본 문서의 두 메커니즘을 구현 후보로 검토하되, 도입 시점과 가드레일을 엄격히 준수한다.

---

## 1. 배경: 리텐션 질문이 위험한 이유

PMF 검증 단계에서 가장 빈번하게 발생하는 표류는 "리텐션이 약해 보일 때 자연스럽게 떠오르는 해법들"이다. 그 해법들은 거의 항상 다음 중 하나로 수렴한다.

- Streak·연속 기록 시스템
- 일일 알림·정기 체크인
- 유저 간 상호작용·커뮤니티
- 캐릭터 애착·게이미피케이션
- 친구 비교·랭킹

이 메커니즘들은 검증된 리텐션 동력이지만, 대부분 **사용자의 의무감·외부 압력·정서적 의존**을 동력으로 삼는다. BlueBird는 이 동력들이 미션(자율성 회복)과 차별화 축(분석가 톤)과 정면 충돌하기 때문에 사용할 수 없다.

따라서 BlueBird의 리텐션 메커니즘은 다음 원칙을 따른다.

> **사용자 간 연결이 아니라, 사용자 자신과의 연결이 깊어지는 메커니즘.**
> 의무감·압력이 아니라, 가치 인식·자산 축적이 동력.

이 원칙 위에서 두 가지 메커니즘을 제안한다.

## 2. 메커니즘 1: 내 사용설명서 (My User Manual)

### 2.1 핵심 컨셉

사용자가 BlueBird를 누적 사용한 결과로, **자기 자신에 대한 매뉴얼**이 만들어진다. 단순한 데이터 누적이 아니라, "당신은 이런 상황에서 이런 패턴이 작동합니다"라는 **자기 운영 매뉴얼** 형태로 가시화된다.

이는 포지셔닝 문서 3절의 "누적 가치 (락인) — 사용할수록 정교해지는 자기 지도"를 구체화한 것이다. 추상적 컨셉이었던 "자기 지도"를 사용자가 인지·소유·갱신할 수 있는 **명시적 산출물**로 전환한다.

### 2.2 작동 메커니즘

사용자가 일정 사용량(예: 30일 또는 100건 입력)을 누적하면, BlueBird가 자동으로 "당신의 사용설명서 v1.0"을 생성한다.

매뉴얼 구성 예시:

**섹션 1: 당신의 활성 인지 왜곡 패턴**
- "당신의 사고에서 가장 자주 작동하는 왜곡: 1위 과잉일반화(38%), 2위 독심술(22%), 3위 파국화(15%)"
- 각 패턴의 대표 사례 3건 (사용자 자신이 입력한 사고에서 추출)

**섹션 2: 트리거 상황 분포**
- "발표·평가 상황에서 과잉일반화 빈도가 평소의 3.2배로 증가합니다"
- "주말보다 일요일 저녁에 독심술 패턴이 1.8배 자주 활성화됩니다"

**섹션 3: 효과 측정 누적**
- "당신은 지난 30일간 47건의 사고를 디버깅했고, 평균 통증 감소 폭은 1.4점입니다"
- "가장 효과적이었던 반증 질문 유형: '반례는 없는가'"

**섹션 4: 다음 30일을 위한 가설**
- "당신의 패턴 분포를 볼 때, 다음 한 달 동안 X 상황에서 Y 패턴이 자주 작동할 가능성이 높습니다"
- "예측 가능한 자기 시스템"의 구체화

매뉴얼은 이후 30일·100건 단위로 v1.1, v1.2로 갱신되며, 사용자는 자기 매뉴얼의 변화 자체를 관찰할 수 있다.

### 2.3 왜 이것이 강한 락인인가

이 메커니즘이 작동하는 심리학적 동력은 **전망이론의 손실 회피**다 (이론 백본 3번, Kahneman & Tversky 1979, λ ≈ 2.25).

사용자가 BlueBird를 떠나는 결정은 단순히 "이 앱을 안 쓴다"가 아니라 **"내 자신에 대한 매뉴얼을 잃는다"** 가 된다. 손실 회피 효과로 인해, 동일 가치의 획득보다 손실이 약 2.25배 강하게 작용한다.

중요한 차별점:
- Streak 손실: 의무 이행 실패감 → 미션과 충돌
- 매뉴얼 손실: 자기 자산 손실 → 자율성 미션과 정합

매뉴얼은 사용자가 **자기 자신을 더 잘 알게 된 결과물**이므로, 그것을 잃는다는 것은 자기 이해 자체를 잃는 것과 등가다. 이는 BlueBird의 미션("자율성 회복")과 충돌하지 않고 오히려 강화한다.

### 2.4 차별화 축과의 정합성

| 차별화 축 | 매뉴얼 메커니즘과의 관계 |
|---|---|
| 분석가 톤 | 강화. "매뉴얼"은 분석가가 자기 시스템을 문서화하는 행위. |
| 신체화 매핑 (보류 중) | 보완. Δstress 데이터가 매뉴얼의 한 섹션이 됨. |
| 자동화 기반 저가 | 정합. 매뉴얼은 누적 데이터 기반 자동 생성, 인간 코치 불필요. |

### 2.5 구현 가드레일

매뉴얼 메커니즘은 잘못 설계하면 **자기 라벨링 강화**나 **결정론적 자기 인식**으로 미끄러질 수 있다. 구현 시 다음을 준수한다.

**(a) 매뉴얼은 "잠정적·수정 가능한 가설"로 제시**
"당신은 X한 사람입니다"가 아니라 "현재까지의 데이터로 볼 때, 당신의 사고에서 X 패턴이 자주 관찰됩니다"의 톤. 사용자를 정의하지 않고, 관찰된 패턴을 보고한다.

**(b) 매뉴얼은 사용자가 수정·반박 가능**
각 섹션에 "이 관찰에 동의하지 않습니다"·"맥락 추가하기" 같은 옵션. 매뉴얼은 BlueBird가 사용자에게 부여하는 정체성이 아니라, 사용자가 검토·수정하는 자기 가설이다.

**(c) 매뉴얼은 부정적 정체성을 강화하지 않음**
"당신은 파국화하는 사람입니다"는 금지. "발표 상황에서 파국화 패턴이 작동했습니다"는 허용. 행동·패턴 수준으로 표현하고, 인격 수준으로 라벨링하지 않는다.

**(d) 매뉴얼 손실 회피를 명시적 마케팅 도구로 쓰지 않음**
"이걸 잃고 싶어요?"·"30일 누적 데이터가 사라집니다" 같은 카피는 의무감 메커니즘으로 전환된다. 매뉴얼의 가치는 사용자가 사용 중 자연스럽게 인지하는 것이지, BlueBird가 압박 도구로 쓰는 것이 아니다.

**(e) 매뉴얼은 export 가능**
사용자가 BlueBird를 떠나기로 결정하면, 매뉴얼을 PDF·텍스트로 내려받을 수 있어야 한다. 이는 자율성 미션과 정합하며, 역설적으로 락인을 강화한다 — "언제든 떠날 수 있다"는 신뢰가 머무는 동력이 된다.

### 2.6 도입 시점

**현재 베타 단계**: 매뉴얼 산출물 생성 자체는 보류. 단, 데이터 스키마는 매뉴얼 생성 가능한 형태로 미리 설계.

**G2 통과 후**: 첫 매뉴얼 산출물(v1.0) 도입. 30일 사용자 또는 100건 입력자에게 자동 생성.

**G3 통과 후**: 매뉴얼 갱신 메커니즘(v1.1, v1.2) 도입. 시계열 변화 가시화.

도입 시점이 베타 후반인 이유: 매뉴얼이 의미 있으려면 **충분한 데이터가 누적**되어야 한다. 30일 미만 사용자에게 매뉴얼을 보여주면 통계적 무의미·낮은 신뢰도로 오히려 가치 인식을 떨어뜨린다.

### 2.7 검증 가설

매뉴얼 메커니즘 도입 후 검증할 가설:

- **H1**: 매뉴얼 v1.0 생성 시점(30일) 직후 1주일 동안 사용 빈도가 유의미하게 증가한다.
- **H2**: 매뉴얼 export 기능을 알고 있는 사용자의 30일 잔존율이, 모르는 사용자보다 낮지 않다 (역설적 락인 가설 검증).
- **H3**: 매뉴얼 생성 사용자의 결제 의향이, 미생성 사용자보다 높다.

H1·H2·H3가 모두 양성이면 매뉴얼은 BlueBird의 핵심 락인 메커니즘으로 확정.

## 3. 메커니즘 2: 익명 통계 맥락화 (Anonymous Statistical Context)

### 3.1 핵심 컨셉

유저 간 상호작용은 거부하되, **다른 사용자들의 익명 통계**를 통해 사용자에게 맥락을 제공한다. "이 패턴이 BlueBird 사용자 중 X%에서 관찰된다"는 형태의 정보 노출.

이는 정서적 공감(다른 사람과의 연결)이 아니라 **분석적 맥락화**(자기 사고를 통계 분포 안에서 위치시킴)다. 사용자에게 "나만 이런 게 아니구나"의 안도감을 주면서도, 다른 사용자와의 직접 접점은 없다.

### 3.2 작동 메커니즘

사용자가 사고를 입력하고 인지 왜곡 패턴이 분류된 직후, 결과 화면에 다음과 같은 통계 맥락이 함께 표시된다.

**예시 1: 패턴 빈도 맥락화**
> "이 사고는 '독심술' 패턴으로 분류되었습니다.
> BlueBird 사용자가 입력한 전체 사고 중 18%가 이 패턴에 해당합니다.
> 30대 인지노동자 그룹에서는 24%로 평균보다 높게 관찰됩니다."

**예시 2: 상황 맥락화**
> "발표 상황에서의 과잉일반화는 BlueBird 사용자 중 67%가 경험합니다.
> 이 패턴은 평균 3.2회 디버깅 후 빈도가 절반으로 감소합니다."

**예시 3: 디버깅 효과 맥락화**
> "당신은 이 사고 디버깅 후 통증이 2점 감소했습니다.
> BlueBird 사용자 평균 감소 폭(1.4점)보다 큰 변화입니다."

### 3.3 왜 이것이 BlueBird에 정합한가

이 메커니즘이 분석가 톤·미션·차별화 축과 정합하는 이유:

**(a) 자기 위치 파악 = 분석적 행위**
"내가 이상한 건 아닌가"는 모든 멘탈헬스 도구 사용자의 공통 질문이다. 일반적 도구는 이를 정서적으로 해소한다("괜찮아요, 누구나 그래요"). BlueBird는 이를 **데이터로 해소**한다 ("18%의 사용자가 같은 패턴을 입력했습니다"). 이는 분석가 톤의 정확한 표현이다.

**(b) 정서적 의존 없는 사회적 안도감**
사용자는 "혼자가 아니다"는 안도감을 얻지만, 그 출처가 다른 사용자와의 정서적 연결이 아니라 **익명 통계**다. 의존·애착이 형성되지 않으면서 고립감은 완화된다.

**(c) 데이터 자산화 가속**
사용자가 입력할수록 BlueBird의 통계 데이터베이스가 풍부해진다. 그 데이터가 다시 다음 사용자에게 더 정교한 맥락을 제공한다. **사용자 수가 증가할수록 모든 사용자의 경험이 개선되는 구조** — 네트워크 효과의 분석적 버전.

이 (c) 항목은 디스턴싱이 만들기 어려운 구조다. 디스턴싱은 인간 코치 1:1 모델이라 사용자 증가가 코치 부담을 늘릴 뿐, 다른 사용자에게 가치가 전이되지 않는다. BlueBird는 자동화·익명 통계 모델이라 이 전이가 가능하다.

### 3.4 차별화 축과의 정합성

| 차별화 축 | 통계 맥락화와의 관계 |
|---|---|
| 분석가 톤 | 강화. "당신의 패턴은 18%에 해당"은 정확히 분석가의 언어. |
| 신체화 매핑 | 보완. Δstress 평균값과의 비교가 가능해짐. |
| 자동화 기반 저가 | 정합. 통계는 자동 집계, 인간 개입 불필요. |

### 3.5 구현 가드레일

통계 맥락화는 잘못 설계하면 **부정적 비교**·**자기 라벨링**·**경쟁심**으로 미끄러질 수 있다. 다음을 준수한다.

**(a) 비교가 아니라 맥락**
"당신은 평균보다 많이 합니다"·"당신은 상위 X%입니다" 같은 순위·평가 표현은 금지. "이 패턴은 X%에서 관찰됩니다"·"평균은 Y점입니다" 같은 분포 정보만 제공.

**(b) 부정적 통계 노출 제한**
"이 패턴 사용자의 50%는 1주일 내 떠납니다" 같은 부정적·낙담형 통계는 금지. 사용자의 행동을 변화시키는 정보가 아니라, 자기 위치를 파악하는 정보만 노출.

**(c) 표본 크기 명시**
30일 미만·100건 미만의 데이터로 통계를 생성하지 않음. 표본이 부족할 때는 "충분한 데이터가 모이면 맥락 정보가 추가됩니다"로 표시. 잘못된 통계로 사용자 인식을 왜곡하지 않음.

**(d) 익명성 절대 보장**
어떤 통계도 개별 사용자를 식별 가능한 형태로 노출되지 않음. 그룹 통계만, 최소 그룹 크기(예: N≥50) 충족 시에만 노출.

**(e) 사용자가 통계 노출을 끌 수 있어야 함**
일부 사용자는 통계 비교가 부담일 수 있다. 설정에서 "통계 맥락 표시 끄기" 옵션 제공. 자율성 미션과 정합.

### 3.6 도입 시점

통계 맥락화는 매뉴얼보다 **데이터 누적이 더 많이 필요**하다. 그룹별 통계가 의미 있으려면 카테고리별로 수백~수천 건의 데이터가 필요하다.

**현재 베타 단계**: 도입 보류. 데이터 수집은 시작.

**G3 통과 이후 또는 사용자 1,000명 누적 시점 (둘 중 늦은 때)**: 첫 통계 맥락 노출 시작. 가장 안정적 통계(전체 패턴 분포)부터 시작해, 점진적으로 세밀한 통계(상황별·그룹별) 추가.

**v2 단계**: 사용자 그룹 비교(예: 30대 인지노동자 vs 전체) 노출. 단, "(b) 부정적 통계 노출 제한" 가드레일 엄격 적용.

### 3.7 검증 가설

- **H4**: 통계 맥락 노출 사용자의 다음 입력 빈도가, 미노출 사용자보다 유의미하게 높다.
- **H5**: 통계 맥락이 사용자의 인터뷰에서 "다른 멘탈케어 앱과 다른 점"으로 언급되는 빈도.
- **H6**: 통계 맥락 노출이 부정적 자기 인식을 강화하지 않는다는 안전성 검증 (인터뷰 정성 분석).

H4·H5가 양성이고 H6에서 안전성이 확인되면 통계 맥락은 v2의 핵심 차별화 도구로 격상.

## 4. 두 메커니즘의 관계

매뉴얼과 통계 맥락은 서로 독립적이지만, 함께 작동할 때 시너지가 발생한다.

| | 매뉴얼 | 통계 맥락 |
|---|---|---|
| 시야 | 자기 자신 (개인 누적) | 다른 사용자들과의 분포 (집단) |
| 시점 | 30일·100건 단위 마일스톤 | 매 사용 시 결과 화면 |
| 동력 | 자기 이해 누적·손실 회피 | 자기 위치 파악·고립감 완화 |
| 데이터 흐름 | 사용자 → 자기 자신 | 전체 사용자 → 개별 사용자 |

매뉴얼이 **수직적 자기 이해**(시간 누적)라면, 통계 맥락은 **수평적 자기 이해**(분포 안에서 위치)다. 두 축이 만나면 사용자는 "나는 시간에 따라 어떻게 변하고 있고, 다른 분석가형 사용자들과 비교해 어디 있는가"를 동시에 알게 된다.

이 결합은 디스턴싱·하루콩·마인들링 어디에도 없는 자기 이해 모델이며, BlueBird의 분석가 톤·자율성 미션과 가장 정합하는 리텐션 동력이다.

## 5. 도입 우선순위

본 문서의 두 메커니즘은 즉시 도입 대상이 아니다. 90일 검증의 핵심 가설을 흐릴 위험이 있어 **단계적 도입**이 필요하다.

### 5.1 베타 단계 (현재)

**도입 항목**: 없음. 데이터 스키마 설계만 진행.

**근거**: 베타 단계의 핵심 가설은 "분석가 톤 + 인지 왜곡 디버깅"의 기본 가치제안 검증. 매뉴얼·통계는 이 가설이 통과된 후 의미.

### 5.2 G2 통과 후 (60일 게이트 통과)

**도입 항목**: 매뉴얼 v1.0 (30일 사용자 대상).

**근거**: G2 통과는 기본 가치제안이 작동한다는 증거. 매뉴얼은 그 위에 락인을 강화하는 메커니즘으로 자연스럽게 추가.

### 5.3 G3 통과 후 또는 사용자 1,000명

**도입 항목**: 통계 맥락화 (전체 분포부터 시작).

**근거**: 통계가 의미 있을 만큼 데이터가 누적된 시점. G3는 외부 진입 시작 시점이라 통계 맥락이 신규 사용자의 가입·잔존에 영향을 줄 수 있음.

### 5.4 v2 단계

**도입 항목**: 매뉴얼 갱신·시계열 비교, 그룹별 통계 비교, 매뉴얼 export.

## 7. 위협 신호 (즉시 멈춤 트리거)

본 문서의 메커니즘들도 잘못 운영되면 본질 위협이 된다. 다음 신호 발생 시 작업 중단:

1. 매뉴얼 카피에 "당신은 X한 사람입니다" 류 인격 라벨링 등장.
2. 통계 노출이 "당신은 평균보다 X합니다" 류 비교·평가 형태로 흐름.
3. 매뉴얼 손실 회피가 BlueBird의 마케팅 메시지로 등장 ("이걸 잃지 마세요" 류).
4. 통계가 부정적·낙담형 정보를 노출 ("이 패턴 사용자는 회복이 어렵습니다" 류).
5. 매뉴얼·통계 메커니즘이 사용자에게 새로운 입력 의무를 요구 (정기 갱신 강제 등).

이 신호들은 모두 "락인 메커니즘이 의무·압력 동력으로 변질"되는 경로다. 발생 시 본 문서로 복귀해 가드레일 재확인.
