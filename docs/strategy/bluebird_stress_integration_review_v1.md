**문서 버전**: v1 (2026-04-26)
**대상 독자**: Claude Code (구현 담당 AI), 파운더
**관련 문서**: `bluebird_mvp_brief_v1.md`, `bluebird_competitive_strategy_v1.md`

## 1. 검토 배경

파운더가 "스트레스 관리 기능을 BlueBird에 포함할 것인가"를 검토했고, 외부 AI(Gemini)가 통합 찬성 의견을 제시했다. 본 문서는 그 의견을 BlueBird의 전략적 맥락(디스턴싱 경쟁 분석, 90일 검증 가설, 차별화 3축, 법적 리스크)에 비추어 재검토한 결과다.

핵심 질문은 표면적으로는 "추가할까 말까"이지만, 실제로는 **"BlueBird를 어떤 카테고리의 도구로 정의할 것인가"** 라는 포지셔닝 결정이다. 이 결정이 흔들리면 90일 검증 데이터의 해석 가능성이 무너진다.

---

## 2. Gemini 제안의 핵심 요약

외부 분석은 다음 네 가지를 제안했다.

1. 스트레스를 "대중적 입구"로 활용해 시장 확장 (TAM 확대)
2. 모닝/이브닝 체크인 시 stress_level 1~5점 입력
3. Δstress 지표 도입 (인지 분석·행동 완료 후 스트레스 변화 측정)
4. 스트레스-왜곡 상관관계 인사이트 ("스트레스 4점 이상일 때 임의적 추론 30% 상승")

부가 질문: "유료 Pro 기능으로 갈 것인가, 기본 기능으로 풀 것인가"

## 3. 검토 결과: 정확히 본 것과 놓친 것

### 3.1 Gemini가 정확히 본 것

**(a) 스트레스와 인지 왜곡의 양방향 관계**
"스트레스는 인지 왜곡의 연료이자 결과물"이라는 판단은 정확하다. 스트레스 상황에서 인지 왜곡 빈도와 강도가 증가하고, 인지 왜곡이 다시 스트레스를 누적시키는 양방향 인과 관계는 CBT 문헌에서 확립된 사실이다.

**(b) 데이터 중심 접근의 정당성**
"명상 음원 만들지 마라, 데이터로 다뤄라"라는 가드레일은 BlueBird 정체성과 정합한다. "힐링 앱으로 오인되는 것을 경계"하라는 경고도 옳다.

**(c) 기술적 저비용 통합**
logs 테이블에 stress_level 필드 추가는 기술적으로 거의 비용 0이다. 기존 인프라 활용 방향성은 적절하다. *(2026-04-28 메모: 그러나 pain_score가 이미 동일 메커니즘으로 작동 중이라 신규 필드는 중복이 됨.)*

### 3.2 Gemini가 결정적으로 놓친 것

**(α) "스트레스 = 시장 확장"이라는 가정의 함정**

Gemini는 "스트레스가 인지 왜곡보다 대중적이라 TAM이 커진다"를 핵심 논거로 제시했다. 이 논리는 표면적으로 매력적이지만, BlueBird의 현재 단계에서는 정확히 잘못된 방향이다.

이유: "스트레스·우울·번아웃 자기관리"라는 넓은 카테고리는 디스턴싱·마인들링·하루콩·인사이드가 이미 점유하고 있다. "스트레스"라는 입구로 진입하는 순간 BlueBird는 이 포화된 시장의 후발주자가 된다.

반면 "인지 왜곡 디버깅"이라는 좁은 입구는 시장이 작아 보이지만, 디스턴싱이 점유하지 않은 분석가 톤 세그먼트가 비어 있다. **검증 단계에서는 "넓은 카테고리의 후발주자"보다 "좁은 카테고리의 1등 후보"가 거의 항상 정답**이다.

Gemini는 한국 멘탈헬스 시장의 경쟁 지형 데이터를 갖고 있지 않아 이 판단을 내릴 수 없었다.

**(β) 검증 가설 분리 원칙의 부재**

Gemini의 마지막 질문 "유료 Pro vs 기본 기능"은 90일 검증 단계에서 시기상조다. 이 질문 자체가 falsifiability 원칙에 대한 이해 부재를 드러낸다.

90일 검증은 단일 가설·단일 변수여야 한다. 스트레스 기능 추가와 동시에 가격 구조 결정까지 진행하면, 90일 후 데이터에서 다음을 분리할 수 없다.

- 리텐션이 인지 왜곡 도구의 효과 때문인가?
- 스트레스 측정 기능 때문인가?
- 가격 구조 변화 때문인가?

검증 단계에서 변수를 늘리는 결정은 데이터 자체를 무효화한다. 이는 본 라운드 브리프의 8절 우선순위 1번(검증 가능성)과 정면 충돌한다.

**(γ) 자기 라벨링의 데이터 노이즈 위험**

stress_level 1~5점 입력을 매일 반복적으로 요구하는 행위는 BlueBird 맥락에서 **"감정적 추론 패턴의 데이터화"** 위험을 만든다. 자기 상태를 숫자로 라벨링하는 행위가 메타인지 훈련이 될 수도 있고, 자기 강화 루프("오늘 스트레스 5점 → 나는 망했다")가 될 수도 있다.

CBT 도구는 다른 도구보다 이 미묘함에 민감해야 한다. BlueBird의 정체성은 사용자가 자기 사고를 더 정확하게 보도록 돕는 것이지, 자기 상태를 더 자주 라벨링하게 만드는 것이 아니다.

## 4. 진짜 분기점: 입력 vs 결과

본 검토에서 발견한 가장 중요한 통찰은, "스트레스를 추가하느냐"가 아니라 **"스트레스를 입력으로 받느냐, 결과로 측정하느냐"** 가 진짜 분기점이라는 점이다.

### 4.1 (a) 스트레스를 입력으로 받는 모델

사용자가 "오늘 스트레스 4점"을 먼저 입력하고, 그것이 인지 왜곡 분석의 트리거가 되는 모델. Gemini가 제안하는 모닝/이브닝 체크인이 여기에 해당.

**효과**: BlueBird를 "스트레스 관리 도구"로 카테고리 이동.
**리스크**: 차별화 축(분석가 톤·인지 왜곡 디버깅) 희석. 디스턴싱·하루콩 등 기존 시장에 후발 진입.
**판단**: 거부.

### 4.2 (b) 스트레스를 결과로 측정하는 모델

인지 왜곡 디버깅 후 "이 사고를 검토한 후 스트레스가 어떻게 변했는가"를 측정하는 모델. Gemini가 제안한 Δstress 지표가 여기에 해당.

**효과**: BlueBird 도구의 효과를 측정하는 메커니즘으로 작동. 차별화 축은 유지.
**리스크**: 거의 없음. 단, 측정 시점·UI 설계가 사용자에게 부담 주지 않아야 함.
**판단**: 채택.

> **2026-04-28 메모:** (b) 모델은 현재 코드에 *이미* 구현되어 있다. `logs.pain_score` (사고 직후) + `intervention.reevaluated_pain_score` (24h 후) → Δpain. PMF plan §6 메트릭에 "평균 Δpain (양수 = 사용자가 효과 보고 있음)"으로 박혀있음. 라벨이 "통증"이냐 "스트레스"냐는 인터뷰 데이터로 결정.

### 4.3 핵심 발견: (b)는 가벼운 신체화 매핑이다

(b) 방식의 "Δstress 측정"은 사실상 신체화-인지 매핑의 텍스트 기반 가벼운 버전이다. 워치·바이오센서 없이도 "주관적 스트레스 지수와 인지 왜곡 패턴의 상관관계"를 데이터화할 수 있다.

이는 별도 검토 중인 신체화 매핑(워치 연동) 결정과 무관하게 BlueBird의 차별화 축을 강화한다. 즉 신체화 매핑이 v2 로드맵으로 미뤄지더라도, (b) 방식의 Δstress는 검증 단계에서 차별화 보강축으로 작동할 수 있다.

## 5. 채택·거부 결정

### 5.1 채택 항목

**(1) Δstress 지표** *(2026-04-28: 이미 Δpain으로 구현됨)*
- 인지 왜곡 디버깅 직전·직후의 주관적 스트레스 변화 측정.
- 사용자가 모듈 A·B·C를 통과하는 흐름 안에서 자연스럽게 측정.
- "디버깅 전 스트레스: ___ / 디버깅 후 스트레스: ___" 형태.
- 90일 검증의 효과 측정 지표 중 하나로 활용.

**(2) 스트레스-왜곡 상관관계 인사이트** *(2026-04-28: 30일+ 사용자 트리거 시 활성화 — `lib/insights/pattern-report.ts` 강화 작업으로 박힘)*
- 누적 데이터 기반으로 "당신의 스트레스가 X점 이상일 때 Y 왜곡 패턴이 증가합니다" 형태의 통찰 제공.
- 사용자에게 "예측 가능한 자기 시스템 지도"를 제공.
- 단, 30일 이상 누적된 사용자에게만 노출. 초기 데이터 부족 시 통계적 무의미.

**(3) 데이터 스키마 추가** — *2026-04-28 거부로 변경*
- 거부 사유: pain_score (0~10)가 이미 동일 메커니즘으로 작동 중. 추가 필드는 중복.
- 재해석: 본 항목은 *원칙*만 채택 — "측정 단위로서의 통증·스트레스 데이터를 기존 인프라로 다룬다."

### 5.2 거부 항목

**(1) 모닝/이브닝 스트레스 체크인**
- 거부 사유: 스트레스를 "입구"로 만드는 설계. 카테고리 이동 발생. 자기 라벨링 노이즈 위험.
- 대안: 스트레스 측정은 인지 왜곡 디버깅 흐름 안에서만 발생.

**(2) 유료 Pro vs 기본 기능 결정**
- 거부 사유: 90일 검증 단계에서 시기상조. 변수 증가로 데이터 해석 가능성 훼손.
- 대안: PMF 검증 후 결정. 본 라운드에서는 모든 사용자에게 동일 노출.

**(3) "스트레스 관리"를 마케팅 카피·랜딩에 노출**
- 거부 사유: 차별화 축(분석가 톤) 직접 충돌. "또 하나의 스트레스 앱"으로 인식 이동.
- 대안: 스트레스는 내부 메커니즘으로만 활용. 외부 메시지는 "인지 왜곡 디버깅" 유지.

## 6. 구현 원칙 (Claude Code가 흔들리지 않기 위해)

본 결정은 미묘한 경계 위에 있다. 구현 중 "이왕 하는 김에" 식 확장이 발생하기 쉽다. 다음 원칙으로 자기 점검한다.

### 6.1 핵심 원칙

> **스트레스는 BlueBird의 입구가 아니라 측정 단위다.**
> 사용자가 "스트레스 받아서 BlueBird 켰다"가 아니라, "사고를 디버깅했더니 스트레스가 줄었다"가 되어야 한다.

이 원칙이 흔들리는 순간 BlueBird는 카테고리 이동을 시작한다. 모든 미시 결정에서 이 원칙을 적용한다.

### 6.2 구현 시 체크리스트

새로운 스트레스 관련 화면·기능·카피를 만들 때마다 다음 질문에 답한다.

1. **이 요소가 사용자를 "스트레스 관리"로 유도하는가, "사고 디버깅의 효과 측정"으로 유도하는가?** 전자라면 재설계.
2. **이 요소가 디스턴싱·하루콩·마인들링에 그대로 붙여도 어색하지 않은가?** 어색하지 않다면 차별화 부족.
3. **이 측정이 사용자에게 새로운 자기 라벨링 부담을 추가하는가?** 추가한다면 흐름 안에 흡수하는 방식 재검토.
4. **이 데이터가 90일 검증 가설을 더 명확히 하는가, 흐리는가?** 흐린다면 거부.

### 6.3 위험 신호 (즉시 멈추고 재검토)

다음이 발생하면 작업을 멈추고 본 문서로 복귀한다.

- 카피에 "스트레스 받으셨나요", "스트레스 관리하기" 같은 표현이 자연스럽게 등장할 때.
- 메인 메뉴·홈 화면에 "스트레스" 단독 진입점이 생성될 때.
- 모닝/이브닝 알림 같은 정기 체크인 메커니즘이 추가될 때.
- 마케팅 카피·랜딩 페이지에 "스트레스"가 핵심 키워드로 등장할 때.

이 신호들은 모두 (a) 모델로의 표류를 의미한다.

## 7. 데이터 활용 가설

채택된 항목들이 90일 검증에서 만들어내는 데이터의 활용 시나리오는 다음과 같다.

### 7.1 도구 효과 입증

Δstress의 평균값이 의미 있게 음수라면 (= 디버깅 후 스트레스가 평균적으로 감소한다면), 이는 "BlueBird가 실제 효과 있다"는 정량적 증거가 된다. 이는 PMF 검증 후 마케팅·투자·임상 협력 단계에서 핵심 자산이 된다.

### 7.2 차별화 축 보강

스트레스-왜곡 상관관계 인사이트는 "사용할수록 자기 데이터가 쌓이고, 그 데이터가 통찰을 만든다"는 BlueBird의 리텐션 가설을 강화한다. 단순한 streak이 아닌 데이터 기반 리텐션 메커니즘.

### 7.3 신체화 매핑 결정의 근거

Δstress 데이터가 의미 있게 작동하면, 신체화 매핑(워치 연동) 도입의 정당성이 데이터로 입증된다. 반대로 Δstress가 의미 없는 노이즈로 나오면, 더 비싼 신체화 매핑은 도입할 가치가 없다는 신호다.

즉 본 결정은 **신체화 매핑의 가치를 저비용으로 사전 검증**하는 효과도 있다.
